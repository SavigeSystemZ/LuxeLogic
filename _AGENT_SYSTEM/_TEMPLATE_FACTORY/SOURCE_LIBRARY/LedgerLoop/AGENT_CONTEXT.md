## 2) Terminology mapping (critical for multi-agent handoff)

The codebase and docs use two overlapping vocabularies. **Always resolve like this:**

| PRD / DATA_MODEL | Current code / DB / API_DESIGN | Notes |
|------------------|--------------------------------|--------|
| **Household** | **Loop** | Top-level tenant container. Same concept. |
| **Membership** (owner/admin/member/viewer) | **LoopMember** (ARCHITECT / CONTRIBUTOR / AUDITOR) | Role sets differ; map Owner→ARCHITECT, Viewer→AUDITOR. |
| **Categories** (with category_groups) | **Buckets** + **category_groups** | Buckets are budget categories; optional `group_id` links to `category_groups`. Both implemented. |
| **Accounts** | **accounts** table, `/:loop_id/accounts` API | Implemented: CRUD, tenant isolation, RBAC (ARCHITECT write, all read). Migration 002_add_accounts.sql. |
| **Payees** | **payees** table, `/:loop_id/payees` API | Implemented: CRUD; optional `payee_id` on transactions. Migration 004_add_payees.sql. |
| **Budget periods / plans** | `budget_periods`, `budget_plans` | Budget engine (M3) is implemented with sheet/envelope APIs. |
| **Envelope / zero-based** | Bucket `current_balance`, `target_amount`, `reset_frequency` | Partial; full envelope math in M3. |

When adding features, prefer **PRD/DATA_MODEL** names in docs and product language; keep **Loop/Bucket/Member** in code/API/DB for consistency with existing implementation.

## 3) Tech stack (current)

- **Runtime:** Node.js (v18+)
- **Language:** TypeScript
- **Backend:** Express 5, REST API
- **DB:** PostgreSQL (Docker Compose), `pg` driver (no ORM)
- **Auth:** JWT (access + refresh), bcrypt passwords
- **Tests:** Jest, Supertest
- **Lint/format:** ESLint, Prettier

**Not yet:** Reconciliation workflow, Prisma/ORM, Auth.js, S3, production Plaid/OFX rollout.

## 4) Implemented vs planned

### Implemented (backend)

- **Auth:** register, login, refresh token (JWT)
- **Loops:** CRUD, get members
- **Members:** invite by email (ARCHITECT only), list members, invite accept flow
- **RBAC:** ARCHITECT / CONTRIBUTOR / AUDITOR; middleware `authorizeRoles`, `authenticateToken`
- **Buckets:** CRUD under `/:loop_id/buckets` (ARCHITECT only in current route); optional `group_id` links buckets to category groups
- **Accounts:** CRUD under `/:loop_id/accounts` (ARCHITECT for create/update/delete, all roles for read)
- **Transactions:** CRUD, splits; transaction logs (audit)
- **Sheet definitions (G.1):** Loop-scoped canonical metadata API for dataset sources, columns, and aggregations (`transactions`, `budget`).
- **Budget engine (M3):** period lifecycle, plan upsert/copy/delete, envelope sheet API (planned/actual/rollover/available).
  - Includes template-seeded period creation support via `template_budget_period_id`.
- **Reports:** summary, bucket balances, business expenses, member contributions
- **Institutions (K):** provider adapters (Plaid/GoCardless/SimpleFIN), secure token bridge registration, ingestion, and pending-review workflows.
- **Agent bridge (L):** read-only tool APIs + local Ollama chat endpoint with tool-call orchestration.
- **Alerts (L.5):** loop-scoped in-memory pub/sub trigger bus + review API + frontend/native notification dispatch.
- **Settings:** `user_settings` table + API (color_scheme, skin)
- **Exchange rates:** service + job (fetch/store rates)
- **Category groups:** CRUD under `/:loop_id/category-groups` (ARCHITECT write, all read); optional `group_id` on buckets.
- **Payees:** CRUD under `/:loop_id/payees` (ARCHITECT write, all read); transactions create/update support optional `payee_id`, and transaction list/detail join `payee_name`.
- **Tests:** basic smoke, auth, rbac, loop, settings, invite accept flow, accounts, category groups, payees, integration.

### Not implemented / partially implemented (from PRD / prompts)

- **Sheet engine (M4):** definitions + dynamic budget column controls are implemented; full multi-sheet grouping/pivot workflow is still partial.
- **Imports (M5):** CSV preview/commit + dedupe/idempotency safeguards are implemented; additional bank-normalization/reconciliation depth remains open.
- **Reports parity:** summary, bucket-balances, and business-expenses support date ranges.
- **Reconciliation:** not started.

## 6) Rules & skills to apply

- **Security:** Tenant isolation (all queries by `loop_id` + membership); no secrets in repo; env vars; RBAC on every sensitive route; audit events for membership/role changes and destructive actions.
- **Data integrity:** Split sum = transaction total; enforce in DB/service; use DB transactions for multi-row writes.
- **API style:** REST; consistent error envelope (code, message, details, correlationId); pagination for lists.
- **Code quality:** TypeScript strict mode; tests for new behavior; RUNBOOK commands must stay valid; follow DEVELOPMENT_STANDARDS.md conventions.
- **Multi-agent coordination:** 
  - Prefer small, mergeable changes; document decisions in ADRs or ARCHITECTURE
  - Keep AGENT_CONTEXT.md and TODO.md updated when you change scope or state
  - Always identify yourself in "Last work" section (8.3)
  - Run tests and lint before finishing work
  - Keep system/prompt/rules snapshot commits on `backup/system-context`; do not merge that branch into `main`
  - See AI_AGENT_COORDINATION.md for detailed protocols
